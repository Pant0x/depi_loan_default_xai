Project Overview

The Automated Explainable AI System is a production-grade machine learning application designed to predict whether a loan applicant is likely to default on their loan. Unlike traditional credit scoring systems that operate as opaque "black boxes," this system places equal emphasis on prediction accuracy and prediction transparency.

The project was developed as part of the DEPI (Digital Egypt Pioneers Initiative) program and addresses a critical gap in the financial technology sector: the need for AI systems whose decisions can be understood, audited, and explained to both technical stakeholders (risk analysts, compliance officers) and non-technical audiences (loan applicants, branch managers).

The system achieves this through a dual-layer approach:

  • Prediction Layer: Multiple machine learning models are trained, tuned, and evaluated on the Kaggle Loan Default dataset. The best-performing model is selected based on rigorous cross-validation and threshold-optimization analysis.
  • Explanation Layer: After each prediction, the system generates human-readable explanations using SHAP (SHapley Additive exPlanations) and LIME (Local Interpretable Model-agnostic Explanations). These explanations break down exactly which applicant features (e.g., high debt-to-income ratio, low credit score) contributed to the model's decision and by how much.

The result is a system where every credit decision is accompanied by a transparent, mathematically grounded justification — enabling fair lending practices and regulatory compliance.

02

ML Pipeline Stages

The project follows a structured, sequential machine learning pipeline organized into four Jupyter Notebooks. Each notebook represents a discrete stage of the data science lifecycle:

Stage 1

Exploratory Data Analysis (EDA)

notebook/EDA.ipynb

The first stage involves loading the raw Kaggle Loan Default dataset and performing a thorough statistical exploration. This includes computing summary statistics (mean, median, standard deviation, quartiles) for every numerical feature, plotting distribution histograms and box plots to identify skewness and outliers, generating a correlation heatmap to detect multicollinearity among features, and analyzing class distribution to determine the degree of imbalance between defaulters and non-defaulters. The insights gathered here directly inform the preprocessing and feature engineering decisions in subsequent stages.

Stage 2

Data Preprocessing

notebook/Preprocessing.ipynb

The preprocessing notebook handles data cleaning and transformation. Operations include handling missing values through imputation strategies (mean/median for numerical, mode for categorical), encoding categorical variables (one-hot encoding for nominal features like employment type, marital status, and loan purpose; ordinal encoding for education level), applying log transformations to right-skewed numerical features (income and loan amount), fitting a StandardScaler to normalize feature distributions to zero mean and unit variance, and splitting the dataset into training, validation, and test partitions with stratified sampling to preserve class ratios. The fitted scaler and feature name list are serialized as scaler.pkl and feature_names.pkl for reuse during inference.

Stage 3

Model Building & Evaluation

notebook/building_models.ipynb

This is the most computationally intensive stage. Multiple classification models are trained and evaluated on the preprocessed data. The models include: Logistic Regression as a linear baseline; Random Forest as an ensemble bagging approach; LightGBM in four configurations (base, SMOTE-augmented for class imbalance, hyperparameter-tuned via Bayesian optimization, and early-stopping with validation monitoring); XGBoost as an alternative gradient boosting framework; an Ensemble Soft Voting Classifier that aggregates predictions across multiple models; and two Tabular Deep Learning MLP architectures built with Keras (a base architecture and an optimized version with dropout regularization and learning rate scheduling). Each model is evaluated using ROC-AUC curves, precision-recall curves, confusion matrices at multiple classification thresholds, learning curve diagnostics to detect overfitting/underfitting, and an accuracy-recall tradeoff analysis. All trained models are serialized to the Models/ directory as .joblib or .keras files.

Stage 4

SHAP & LIME Explainability

notebook/shap_lime_explainer.ipynb

The final notebook applies post-hoc explainability techniques to the best-performing model. SHAP TreeExplainer is used to compute Shapley values for every feature in every prediction, generating three types of visualizations: waterfall plots (showing how each feature shifts the prediction from the base value for a single applicant), beeswarm plots (showing the global distribution of feature impacts across the entire dataset), and feature importance bar charts (ranking features by their mean absolute SHAP value). In parallel, LIME is applied to generate local linear surrogate models around individual predictions, providing an independent cross-validation of which features the model considers most important for each specific applicant. The generated plots are saved to the images/ directory.

03

Feature Engineering

The system automatically transforms raw applicant inputs into a 36-dimensional feature space that the trained models expect. This transformation happens in real-time during inference and includes the following categories of engineered features:

Binary & Ordinal Encoding

Yes/No categorical fields (e.g., HasMortgage, HasDependents, HasCoSigner) are mapped to 1/0 integer values. Education level is ordinally encoded as: High School → 0, Bachelor's → 1, Master's → 2, PhD → 3.

One-Hot Encoding

Nominal categorical features (EmploymentType, MaritalStatus, LoanPurpose) are one-hot encoded with reference classes dropped to avoid multicollinearity. The dropped reference categories are: EmploymentType_Full-time, MaritalStatus_Divorced, and LoanPurpose_Auto.

Log Transformations

Income and LoanAmount are log-transformed using ln(x + 1) to reduce right-skewness and normalize their distributions. A derived feature log_loan_to_income is computed as the difference of these log values.

Financial Ratios

Several ratio-based features are computed: loan_to_income (loan amount divided by income), monthly_income (annual income / 12), employment_ratio (months employed relative to age), monthly_payment_est (estimated monthly payment from loan terms and interest rate), and payment_to_income (monthly payment relative to monthly income).

Risk Composite Scores

A weighted risk_score is computed as: 0.40 × (1 − normalized_credit_score) + 0.35 × DTI_ratio + 0.25 × normalized_interest_rate. A debt_burden_score combines DTI ratio with interest rate multiplicatively.

Interaction & Flag Features

The age_employment_interaction captures the product of age and employment duration. Binary high_risk_flag and is_very_high_risk flags trigger when combinations of poor credit score, high DTI ratio, and elevated interest rate exceed predefined thresholds.

Model Training & Evaluation

The system trains and serializes 12 model artifacts across 7 distinct algorithm families. Each model addresses different aspects of the loan default prediction problem:

Model Type Description
lightgbm_model (1).joblib Gradient Boosting Base LightGBM classifier with default hyperparameters. Serves as the primary tree-based model for SHAP explainability due to native TreeExplainer support.
lightgbm_smote_model.joblib Gradient Boosting LightGBM trained on SMOTE-augmented data to address class imbalance. Synthetic minority oversampling equalizes defaulter/non-defaulter representation during training.
lightgbm_tuned_model.joblib Gradient Boosting LightGBM with hyperparameters optimized via Bayesian search (Optuna or similar). Tuned parameters include learning rate, max depth, num_leaves, min_child_samples, and regularization terms.
lightgbm_val_earlystop_model (1).joblib Gradient Boosting LightGBM trained with early stopping on a held-out validation set. Training halts when validation loss stops improving, preventing overfitting.
xgboost_model (1).joblib Gradient Boosting XGBoost classifier providing an alternative gradient boosting implementation. Uses scale-invariant tree splitting for robust handling of unscaled features.
random_forest_model.joblib Ensemble (Bagging) Random Forest classifier using bootstrap-aggregated decision trees. Provides feature importance rankings through mean decrease in impurity.
logistic_regression_pipeline.joblib Linear Model Logistic Regression with internally pre-packaged StandardScaler. Serves as the interpretable linear baseline — coefficients directly indicate feature direction and magnitude.
ensemble_soft_voting_model.joblib Ensemble (Voting) Soft voting classifier that aggregates predicted probabilities from multiple constituent models and averages them for the final prediction.
tabular_dl_model.keras Deep Learning Base Tabular MLP (Multi-Layer Perceptron) implemented in Keras. Uses dense layers with ReLU activations for non-linear feature interaction learning.
optimized_tabular_mlp.keras Deep Learning Optimized MLP with dropout regularization, batch normalization, learning rate scheduling, and early stopping callbacks to reduce overfitting and improve generalization.

Evaluation Metrics

All models are evaluated using the following metrics and diagnostic visualizations:

  • ROC-AUC Curves: Receiver Operating Characteristic curves plotting True Positive Rate against False Positive Rate at every classification threshold. The area under this curve (AUC) measures overall discriminative ability — a perfect classifier achieves AUC = 1.0, while random guessing yields AUC = 0.5.
  • Precision-Recall Curves: Particularly important for imbalanced datasets where the positive class (defaulters) is underrepresented. These curves plot Precision (of predicted defaulters, how many actually defaulted) against Recall (of actual defaulters, how many were correctly identified) at each threshold.
  • Confusion Matrices: 2×2 matrices showing True Positives, True Negatives, False Positives, and False Negatives at selected classification thresholds. The system generates confusion matrices for all models simultaneously for side-by-side comparison, as well as best-model threshold comparison matrices.
  • Learning Curve Diagnostics: Plots of training and validation loss/accuracy as a function of training set size. These diagnose whether a model suffers from high bias (underfitting, indicated by converging high error) or high variance (overfitting, indicated by diverging training/validation curves).
  • Accuracy-Recall Tradeoff: Analysis of how adjusting the classification threshold affects the balance between overall accuracy and default recall. A lower threshold catches more defaulters (higher recall) but increases false positives (lower precision).
05

Explainability (SHAP & LIME)

The core innovation of this system is its commitment to transparent, interpretable predictions. Every credit decision is accompanied by visual and textual explanations generated through two complementary frameworks:

SHAP (SHapley Additive exPlanations)

Game Theory

SHAP is rooted in cooperative game theory. It treats each feature as a "player" in a cooperative game where the "payout" is the model's prediction. The Shapley value for each feature represents its fair contribution to the prediction — computed by averaging the feature's marginal contribution across all possible orderings of features.

For tree-based models (LightGBM, XGBoost), the system uses TreeExplainer, which computes exact Shapley values in polynomial time by exploiting the tree structure. This is orders of magnitude faster than the kernel-based approximation required for arbitrary models.

SHAP Visualization Types:

  • Waterfall Plot: Shows a single applicant's prediction decomposition. Starting from the base value (average model output across the training set), each feature's SHAP value is displayed as a red (increasing default risk) or blue (decreasing default risk) bar, building up to the final prediction value. This lets a loan officer see exactly why a specific applicant was flagged.
  • Beeswarm Plot: A global overview showing every sample in the dataset as a dot, colored by feature value (red = high, blue = low), positioned by SHAP impact. This reveals both which features are globally important and the direction of their effects.
  • Feature Importance Bar Chart: Ranks features by their mean absolute SHAP value across all predictions. This provides a simple, intuitive ranking of which features matter most to the model overall.

LIME (Local Interpretable Model-agnostic Explanations)

Local Surrogacy

LIME operates on a fundamentally different principle from SHAP. Rather than computing exact game-theoretic attributions, LIME builds a simple, interpretable model (typically a linear regression or decision tree) that approximates the complex model's behavior in the local neighborhood of a specific prediction.

The LIME process works as follows: (1) The applicant's feature values are perturbed by sampling random variations. (2) Each perturbed sample is fed through the trained model to obtain predictions. (3) The perturbed samples are weighted by their proximity to the original applicant (closer samples receive higher weights). (4) A weighted linear model is fit to the (perturbation, prediction) pairs. (5) The coefficients of this linear model indicate which features most influenced the prediction locally.

Why Both SHAP and LIME?

  • Cross-Validation: If SHAP and LIME agree on the most important features for a prediction, confidence in the explanation is high. If they disagree, it may indicate the model's decision boundary is complex in that region and warrants manual review.
  • Complementary Perspectives: SHAP provides exact, theoretically grounded attributions. LIME provides intuitive, model-agnostic approximations. Together, they offer a more complete picture than either alone.
  • Regulatory Compliance: Financial regulators increasingly require that AI-driven credit decisions be explainable. Providing two independent explanation methods strengthens the audit trail.

Deployment Architecture

The production system is deployed as a stateless, decoupled multi-service application on Render. The architecture separates concerns between data processing/ML inference and user interface presentation:

Client Layer
🌐

Web Browser

User fills out the loan application form with applicant details (age, income, credit score, loan amount, etc.)

↓
Presentation Layer
🎨

Frontend Web (Flask)

Renders the application form and risk dashboard. Converts form inputs into JSON payloads and forwards them to the Backend API. Displays returned SHAP charts, LIME charts, and risk level indicators.

↓
Inference Layer
⚡

Backend API (FastAPI)

Receives JSON payloads, applies feature engineering (27 computed features), scales inputs with the fitted StandardScaler, runs model inference on the selected classifier, generates SHAP waterfall plots and LIME explanations in-memory, and returns base64-encoded charts with prediction probabilities.

↓
ML Artifacts
scaler.pkl StandardScaler
feature_names.pkl 36 Feature Names
lightgbm_model.joblib LightGBM Classifier
xgboost_model.joblib XGBoost Classifier

Key Design Decisions

  • Stateless Architecture: No database is required for the core prediction workflow. All feature engineering, scaling, inference, and visualization generation happen entirely in-memory during each request. This simplifies deployment, eliminates state synchronization issues, and allows horizontal scaling.
  • Separated Services: The Flask frontend and FastAPI backend are deployed as independent Render Web Services from the same GitHub repository (using different root directories). This allows independent scaling — the compute-heavy backend can be allocated more resources without affecting the lightweight frontend.
  • Human-Readable SHAP Labels: SHAP waterfall charts use raw, unscaled feature values (e.g., "Income = $55,000") rather than standardized Z-scores as axis labels. This makes the charts immediately understandable to non-technical loan officers without requiring statistical knowledge.
  • Graceful Degradation: If the SHAP or LIME visualization generation fails for a particular input (e.g., edge-case feature values), the system still returns the numerical prediction and risk level, displaying a fallback message in place of the chart.

  • XGBoost — Alternative gradient boosting with scale-invariant tree splitting
  • Keras / TensorFlow — Tabular deep learning MLP architectures
  • imbalanced-learn (SMOTE) — Synthetic minority oversampling for class imbalance
  • pandas / NumPy — Data manipulation and numerical computation
  • joblib / pickle — Model serialization and artifact persistence

Explainability Frameworks

  • SHAP — TreeExplainer for exact Shapley value computation on tree-based models
  • LIME — Local surrogate model fitting for model-agnostic per-prediction explanations
  • matplotlib — Visualization backend for generating SHAP waterfall, beeswarm, and bar plots

Web Application & API

  • FastAPI — Asynchronous Python API framework for ML inference endpoints
  • Flask — Lightweight web framework for the frontend dashboard
  • Uvicorn — ASGI server for FastAPI in production
  • Gunicorn — WSGI server for Flask in production
  • Pydantic v2 — Request/response schema validation

Infrastructure & Deployment
